# Банковский RAG-консультант

MVP-проект Retrieval-Augmented Generation (RAG) для «банковского консультанта», который отвечает на вопросы клиентов по кредитам, вкладам, ипотеке и общим условиям обслуживания на основе внутренней базы документов банка.

Автор: Д.Касьяненко

## Цели проекта

- Построить простой, но воспроизводимый RAG-пайплайн для русскоязычных банковских продуктов.  
- Сравнить три подхода к retrieval: similarity search, MMR и BM25, а также гибридный поиск.  
- Показать, как оценивать retriever (HitRate@4, MRR) и качество RAG-ответов с помощью простых метрик overlapping-токенов.  
- Продемонстрировать оптимизацию производительности retrieval (базовый vs оптимизированный режим).

## Данные и домен

В качестве «знаний» используются текстовые `.md`‑файлы с информацией о продуктах банка:

- Кредиты (`*.md`, product_type = `credit`).  
- Вклады (`*.md`, product_type = `deposit`).  
- Ипотека (`*.md`, product_type = `mortgage`).  
- Общие вопросы (`FAQ.md`, product_type = `general`).

Каждый документ нормализуется (очистка Markdown-разметки, пробелов и т.п.), затем разбивается на **чанки**. Для каждого чанка в метаданных сохраняются: `source` (имя файла), `product_type`, `chunk_id`, `chunk_len` и др.

## Архитектура и стек

### Библиотеки

- Python, `pandas`, `numpy`, `matplotlib`, `seaborn`.  
- `langchain`.  
- `chromadb`.  
- `rank-bm25`.  
- `transformers`, `torch`.

### Компоненты пайплайна

1. **Предобработка и разбиение текста**  
   - Загрузка `.md`‑файлов из каталога `data/raw/documents`.  
   - Функции `normalize_text` и `detect_product_type` для очистки и маркировки.  
   - Сравнение трёх стратегий разбиения: `CharacterTextSplitter`, `NLTKTextSplitter`, `RecursiveCharacterTextSplitter`.  
   - Выбор итоговой стратегии (используется `recursive` с размером чанка ≈ 350 символов и overlap 60).

2. **Векторизация и хранилище**  
   - Эмбеддинги: `intfloat/multilingual-e5-large` (fallback: `intfloat/multilingual-e5-base`).  
   - Векторное хранилище: `Chroma` с указанием `persist_directory` (`./chromadb`).  
   - Векторизация всех финальных чанков и сохранение коллекции `bank_consultant_kb`.

3. **Retrievers**  
   - `similarityretriever` (dense similarity, top‑k).  
   - `mmrretriever` (Maximal Marginal Relevance, `k=4`, `fetch_k=8`, `lambda_mult=0.5`).  
   - Кастомный `BM25RetrieverSimple` поверх `rank-bm25`.  
   - **Гибридный retriever** `hybrid_search`:
     - отдельный вызов dense similarity и BM25;  
     - объединение результатов по `chunk_id`;  
     - score = α·(1/vec_rank) + (1−α)·(1/bm_rank), по умолчанию α = 0.6.
   - Опциональная компрессия контекста `compress_document`: выбор нескольких предложений из чанка по пересечению токенов с запросом.

4. **Advanced retrieval (оптимизированный режим)**  
   - Функция `advanced_retrieve` включает:
     - грубую классификацию `product_type` по тексту запроса;  
     - генерацию нескольких перефразированных запросов `generate_multiqueries`;  
     - запуск гибридного поиска для всех вариантов;  
     - reranking по пересечению токенов запроса и контекста (`rerank_by_overlap`);  
     - LRU-кэш на уровне retrieval для ускорения повторных запросов.

5. **LLM и генерация ответа**  
   - Модель: `Qwen/Qwen2.5-1.5B-Instruct` через `transformers.pipeline("text-generation", ...)`.  
   - Жёсткий system prompt с инструкцией: отвечать по банковским документам, быть самостоятельным консультантом, не фантазировать вне контекста и явно ссылаться на источники.  
   - Функция `build_context(docs)` для склейки выбранных чанков в единый контекст с указанием `source` и `chunk_id`.  
   - Функция `answer_with_rag`:
     - получает документы через `advanced_retrieve`;  
     - строит контекст и формирует user‑сообщение (`query + context`);  
     - вызывает Qwen; при ошибке — fallback-ответ из самих документов;  
     - возвращает словарь с полями `answer`, `sources`, `context`.

## Оценка качества

### Оценка retrieval

- Формируется таблица `eval_questions`: для 20 запросов вручную заданы «золотые» чанки (`gold_chunks`).  
- Для каждого метода (`similarity`, `mmr`, `bm25`, `hybrid`, `hybrid_compressed`) считаются:
  - **HitRate@4** — есть ли среди топ‑4 хотя бы один «золотой» чанк.  
  - **MRR** (Mean Reciprocal Rank) по позициям «золотых» чанков.

Полученная таблица:

| method            | HitRate@4 | MRR   |
|-------------------|-----------|-------|
| similarity        | 1.00      | 0.950 |
| mmr               | 1.00      | 0.950 |
| hybrid            | 1.00      | 0.942 |
| hybrid_compressed | 1.00      | 0.942 |
| bm25              | 0.95      | 0.858 |


### Оценка RAG‑ответов

- Таблица `qa_eval` с 5 вопросами и референсными краткими ответами (например, «минимальный ежемесячный платёж», «минимальная сумма вклада», «максимальная сумма ипотеки» и т.п.).  
- Для каждого вопроса вызывается `answer_with_rag`, и считаются прокси‑метрики на основе Jaccard-пересечения токенов:
  - `faithfulness_proxy` — пересечение между ответом и контекстом;  
  - `answer_relevancy_proxy` — пересечение между ответом и референсным ответом;  
  - `context_relevancy_proxy` — пересечение между контекстом и референсным ответом.

Результаты собираются в `quality_df`.

### Оценка производительности

- Функция `timed_run(queries, use_cache)` сравнивает:
  - **baseline**: многократный вызов `advanced_retrieve` без кэша;  
  - **optimized**: тот же набор запросов, но с LRU-кэшем.  
- Считаются `avg_time_sec`, `median_time_sec`, `p95_time_sec` и выводятся как таблица:

| mode      | avg_time_sec | median_time_sec | p95_time_sec |
|----------|--------------|-----------------|-------------|
| baseline | 0.1265       | 0.1201          | 0.1909      |
| optimized| 0.0416       | 0.0000          | 0.1443      |
